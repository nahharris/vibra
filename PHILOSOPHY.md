# Vibra Philosophy

Vibra is a programming language for code that is mostly written, reviewed, and
rewritten by LLMs. Human readability matters, but it is not the primary design
constraint. The language should make the correct path obvious to an agent with
limited context, and it should make unsafe or ambiguous choices hard to express.

This document is a decision contract. When language design, standard library,
runtime, or tooling choices conflict, prefer the rule that gives LLMs fewer
degrees of freedom, more local context, and safer defaults.

## Decision Order

1. Optimize for LLM-authored code.
2. Prefer one canonical way to express each idea.
3. Make unsafe behavior explicit and guarded.
4. Require intent to be written down in code.
5. Build tooling that guides agents before runtime failure.

## LLMs Are The Primary Authors

Vibra should assume source code is generated by agents. Syntax, diagnostics,
schemas, examples, and standard library APIs should be designed for an LLM that
needs to predict the next valid node in a structured document.

Prefer:

- Regular YAML shapes over clever syntax.
- Reserved `$` keys for language forms and qualified calls.
- Stable schemas and diagnostics that tools can feed back into agents.
- Explicit imports and names that make local context sufficient.
- Small, typed APIs with visible argument names.

Avoid designs that depend on taste, convention, hidden global state, or several
equally valid spellings. Every extra choice is another chance for hallucination.

## One-Way Code

If two forms mean the same thing, one should become canonical and the other
should be rejected or linted. Vibra should reduce equivalent spellings even when
humans might find alternatives convenient.

Language and tooling should favor:

- One invocation shape for intrinsics, user functions, and imports.
- One statement sequencing form in each context.
- One canonical formatting style.
- One annotation spelling, currently `=`-prefixed annotations such as `=where`
  and `=doc`.
- One import model, with public symbols defined by top-level names and private
  symbols marked by a leading `-`.

Compatibility should not preserve ambiguity indefinitely. Migration tools are
better than permanent alternate syntaxes.

## Secure By Nature

LLMs make mistakes, so safety cannot depend on agents remembering every rule.
The safe path should be the default path.

Vibra should prefer:

- Default-deny access to privileged host capabilities.
- Runtime-minted grants and `$capability` types for authority.
- Narrowable permissions that can only reduce authority, not widen it.
- Nominal wrappers and interfaces that make invalid states unrepresentable.
- Typed `result` and `option` flows instead of unchecked failure paths.
- Clear diagnostics when code crosses a trust boundary.

Unsafe escape hatches must be visible in source and easy for tooling to flag.
The current arbitrary `$wasm` escape hatch is a known gap; future design should
make `$wasm` trusted-stdlib-only or require an explicit unsafe/trust policy.

## Explicit Intent

Agents should leave their assumptions in the program. If an intent matters for
typechecking, security, dispatch, or API behavior, it should be written
explicitly instead of inferred from context.

Prefer:

- Explicit `args` and `return` on functions.
- Explicit generic type arguments at call sites and type positions.
- Explicit `$cast` for nominal wrapper boundaries.
- Explicit `$match` arms with local bindings.
- Explicit grant arguments for privileged APIs.
- Explicit implementation blocks for nominal interface satisfaction.

Inference is useful only when it narrows a local expression without hiding a
cross-module, security, or type-contract decision.

## Tooling Is Part Of The Language

Vibra tooling should not be an afterthought. For an LLM-first language, the
formatter, linter, schemas, diagnostics, and query APIs are part of the safety
model.

Tooling should provide:

- Check-only formatting by default, with mutation behind an explicit `--write`.
- Structured diagnostics with stable codes, spans, severity, and fixes.
- YAML-first output for agent workflows, with JSON and SARIF as compatibility
  formats.
- Schemas for expressions, types, diagnostics, manifests, and query responses.
- Context queries that tell an agent which keys, types, imports, and symbols are
  valid at a position.

A good Vibra tool should guide an agent toward valid code before the compiler or
runtime has to reject it.

## When In Doubt

Choose the design that is easiest for an LLM to complete correctly from local
context. Prefer explicit structure over cleverness, guardrails over trust,
canonical forms over personal style, and typed authority over ambient power.
